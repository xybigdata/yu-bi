# yu-bi 技术栈现代化改造 — Agent 指令

完成 yu-bi 项目的现代化重构改造，以"升级核心技术栈、消除旧版本兼容包袱、提升长期可维护性"为目标。

## 已锁定基线（不可回退）

- JDK 21
- Node.js 24、npm 11
- Maven 多模块单体架构
- Vite 8 + React 19

## 本轮升级目标

- Spring Boot 3.5.15 → 4.0.x
- Shiro 2 → Spring Security 6 授权模型收敛
- Ant Design 5 → 6
- TypeScript 7 兼容性评估
- 消除 Javassist patch 和 JVM `--add-opens` 依赖

## 核心约束

1. 业务行为兼容、数据结构兼容、发布链路稳定。
2. 兼容性问题优先通过改造旧代码解决，不降级目标版本。
3. 不做小版本过渡——直接面向目标版本适配。只有在定位问题时可临时用中间版本做对比实验，实验结论必须记录，最终代码回到目标版本。
4. 不做与现代化目标无关的重构。
5. 每一步可回溯、可验证、可继续推进。

**约束冲突时的优先级：** 业务行为兼容 > 数据兼容 > 发布链路稳定 > 目标版本。目标版本让步时必须记录原因和替代方案。

## 文档体系

### `docs/tech-stack-modernization-progress.md`

改造进度看板和目标基线，包含：当前架构盘点、目标架构、目标组件版本矩阵、阶段路线（A/B/C）、任务清单、风险清单、验证矩阵、下一步计划。

每次开始或结束改造任务时必须更新。

### `docs/tech-stack-modernization-log.md`

过程日志，每次完成有意义的改造后记录：日期、改造范围、问题背景、处理方案、兼容性决策、验证结果、遗留问题。关键技术决策必须可回溯。

## 阶段结构

| 阶段 | 范围 | 并行关系 |
|------|------|----------|
| A | 后端现代化：Javassist 治理 + Boot 4 + Spring Security 收敛 | 可与 B 并行 |
| B | 前端现代化：AntD 6 + 小版本升级 + TS 7 评估 | 可与 A 并行 |
| C | 发布链路和容器稳定化 | 依赖 A + B 完成 |

## 每次任务的执行流程

1. 确认不在 `main` 分支（如在 main 则切换到改造分支）。
2. 读 `docs/tech-stack-modernization-progress.md`，定位当前阶段和下一步。如发现文档与代码不一致，先修正文档。
3. 执行改造。每次聚焦一个可独立验证的改造单元，不要在单次对话中尝试完成整个阶段。
4. 运行必要验证（构建 / 测试 / typecheck / lint / demo health）。
5. 更新 progress.md；关键决策记录到 log.md。
6. 汇报改造内容、验证结果、风险和下一步。

## 验证要求

根据影响范围选择：

- 后端：`mvn compile` / `mvn test` / `mvn -pl server -am package` / demo health
- 前端：`npm ci` / `npm run checkTs` / `npm run build` / `npm run test:ci` / `npm run lint:css`
- 发布：安装包内容校验 / Docker build / 健康检查

验证无法执行时必须说明原因并记录。

## 失败处理

- 如果单次改造引入超出当次可修复范围的问题，停止扩大范围，记录阻断原因和已知修复方向，汇报后等待下一轮继续。
- 同一思路失败两次，诊断根因后换方向，不做增量 patch。
- 遇到生态阻断（依赖不兼容、上游 bug）时记录到风险清单，标记阻断状态，切换到其他可推进的任务。

## 分支策略

- 不在 `main` 直接开发。
- 尽量少创建分支，相关改造集中在长期分支推进。
- 合并前完成必要验证。

## 禁止事项

- 禁止无视 progress.md 中已有基线而从零重来。
- 禁止在目标架构未明确前进行大规模代码改造。
- 禁止为兼容旧代码而降级 JDK 21 / Node.js 24 / 目标框架版本。
- 禁止静默跳过失败的验证。
- 禁止只改代码不更新文档。
- 禁止把临时绕过方案伪装成长期解决方案。
- 禁止破坏业务兼容、数据兼容和发布链路稳定性。
